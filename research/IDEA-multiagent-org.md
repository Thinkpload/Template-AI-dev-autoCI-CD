# Идея: Мультиагентная организация

> Статус: raw idea / to explore → **внедрить в wizard**
> Дата: 2026-03-28

---

## Суть идеи

Вместо одного "умного агента" — построить **виртуальную организацию** из специализированных агентов с чёткими ролями, объединённых общей целью. Пользователь не управляет каждым агентом вручную — он выступает **советом директоров**: задаёт стратегию, приоритеты, бюджет внимания. Оперативное управление берёт на себя **CEO-агент**, который координирует команду.

Аналог из мира ПО: [agency-agents](https://github.com/msitarzewski/agency-agents) — фреймворк с жёсткими ролевыми контрактами.

---

## Почему это важно

Обычный подход "притворись старшим разработчиком и сделай всё" имеет потолок:

- Один агент несёт слишком широкий контекст → деградирует на сложных задачах
- Нет специализации → посредственное качество в каждой роли
- Нет разделения ответственности → сложно дебажить что пошло не так

Мультиагентная организация решает это через **контракты ролей**: каждый агент знает свою зону ответственности и интерфейс взаимодействия с другими.

---

## Структура организации (черновик)

```
Совет директоров (пользователь)
        │
        ▼
      CEO-агент
     /    |    \
    /     |     \
  Dev   QA    PM
  Lead  Lead  Lead
  / \         |
Dev  Dev    Writer
```

### Роли и зоны ответственности

| Роль        | Отвечает за                                            | Инструменты    |
| ----------- | ------------------------------------------------------ | -------------- |
| CEO         | Координация, декомпозиция задачи, итоговый результат   | Все агенты     |
| PM          | Требования, приоритизация, acceptance criteria         | Документы, ТЗ  |
| Dev Lead    | Архитектура, code review, решение технических блокеров | Код, Git       |
| Dev         | Имплементация фич                                      | Код, тесты     |
| QA Lead     | Тест-план, валидация результата                        | Тесты, браузер |
| Tech Writer | Документация, README, changelog                        | Документы      |

## Доказанные кейсы (личный опыт)

### Кейс 1: WordPress-сайт на VPS

**Задача:** поднять сайт с нуля под бизнес-задачу.

**Что сделали агенты:**

- Получили SSH-доступ к VPS, к которому пользователь уже прикрутил домен
- Развернули WordPress
- Кастомизировали дизайн под требования
- Создали кастомные post types и кастомные поля (ACF или аналог) под бизнес-логику
- Настроили автоматический постинг SEO-текстов, которые сами же и генерируют
- Умеют работать двумя способами: через WP REST API программно **и** через браузер (кликают кнопки в админке)

**Вывод:** агенты способны работать с реальной инфраструктурой от нуля до рабочего продукта без постоянного ручного контроля.

---

### Кейс 2: Комплексная документация веб-проекта

**Задача:** задокументировать существующий проект веб-разработки.

**Источники, с которыми работали агенты:**

- ТЗ в Google Docs (основное ТЗ)
- ТЗ для покупного решения в Google Docs
- Сайт по ссылке (просмотр живого продукта)
- GitLab-репозиторий — подключились, проанализировали **все** исходники

**Результат:** комплексный документ, синтезирующий все источники в единую картину.

**Вывод:** агенты способны агрегировать разрозненные источники знаний (доки, веб, код) и создавать из них структурированный артефакт.

## Ключевое отличие от "притворись экспертом"

| Подход                            | Проблема                                                                       |
| --------------------------------- | ------------------------------------------------------------------------------ |
| `Представь что ты синьор-помидор` | Агент пытается быть всем сразу → размытый контекст, нет реальной специализации |
| Роль с контрактом                 | Агент знает: что он делает, чего НЕ делает, кому передаёт результат            |

Хороший контракт роли описывает:

1. **Цель** — зачем эта роль существует
2. **Вход** — что принимает (артефакты, запросы)
3. **Выход** — что производит
4. **Границы** — что НЕ входит в зону ответственности
5. **Эскалация** — когда и кому передавать блокер

---

## Следующий эксперимент

Собрать команду агентов чисто под **разработку программного проекта**. Минимальный состав:

- CEO (оркестратор)
- PM (требования → stories)
- Dev Lead (архитектура + review)
- Dev × 2 (имплементация)
- QA (тесты + валидация)

Проверить: насколько чёткое ролевое разделение улучшает качество vs монолитный подход.

## Paperclip — главный референс

**[paperclipai/paperclip](https://github.com/paperclipai/paperclip)** — open-source оркестрация для "компаний без людей" (⭐ 36k+)

> _"If OpenClaw is an employee, Paperclip is the company"_

Node.js сервер + React UI, который:

- Управляет командой агентов через org chart (иерархия, как в реальной компании)
- Каждый агент знает миссию компании — все задачи трейсятся к общей цели
- **Heartbeats** — агенты просыпаются по расписанию, проверяют работу, действуют
- **Cost control** — бюджеты на агентов, мониторинг трат
- Работает с любым агентом: Claude Code, Codex, Cursor, OpenClaw, bash-скрипты, HTTP
- **Clipmart** (coming soon) — маркетплейс готовых "компаний" с одним кликом

### Флоу в Paperclip

1. Задай цель: _"Build the #1 AI note-taking app to $1M MRR"_
2. Найми команду: CEO, CTO, engineers, designers, marketers — любой бот, любой провайдер
3. Утверди стратегию, выстави бюджеты, запусти → мониторь с дашборда (в т.ч. с телефона)

### Почему это точное совпадение с идеей

- Пользователь = совет директоров (approve strategy, set budgets)
- CEO-агент = оперативное управление командой
- Чёткие роли с goal alignment, не "притворись экспертом"
- Можно управлять с телефона пока агенты работают 24/7

## Интеграция в wizard этого шаблона

**Задача:** внедрить концепцию мультиагентной организации в wizard проекта.

Что это означает на практике:

- Wizard должен уметь конфигурировать не одного агента, а **команду агентов с ролями**
- На выходе wizard генерирует не просто проект, а **org chart агентов** с контрактами ролей
- CEO-агент получает миссию проекта и декомпозирует её на команду
- Интеграция с Paperclip как оркестратором (или реализация аналогичного паттерна внутри BMAD)
