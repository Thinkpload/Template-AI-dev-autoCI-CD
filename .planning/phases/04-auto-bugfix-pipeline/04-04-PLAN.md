  <how-to-verify>
    Work through each check in order. Each check requires a real GitHub push or Claude Code session.

    ---

    **Check 1 — CI-01: Security Audit step fails on high vulnerabilities**

    Option A (safe, non-destructive): Confirm the step is wired correctly by reading ci.yml:
    ```bash
    grep -A3 "Security Audit" .github/workflows/ci.yml
    ```
    Expected: `npm audit --audit-level=high > audit.log 2>&1 || (cat audit.log && exit 1)`

    Option B (full end-to-end): Push a branch with a known-vulnerable package temporarily added to package.json, trigger a CI run, verify the Security Audit step fails with a non-zero exit code in the Actions UI.

    ---

    **Check 2 — CI-02: Structured issue creation**

    Trigger a CI failure by temporarily breaking a test (e.g., change `expect(true).toBe(true)` to `expect(true).toBe(false)` in any wizard test), push to a branch, and let CI run.

    Verify in GitHub Issues that a new issue was opened containing:
    - `**Job:**` field with the job name
    - `**Branch:**` field with the branch name
    - `**Commit:**` field with the commit SHA (40-character hex)
    - `**Run:**` field with a link to the Actions run
    - A code block with the test failure log excerpt

    Revert the broken test after confirming.

    ---

    **Check 3 — CI-03: /fix-issue slash command fetches issue and opens PR**

    Open Claude Code in this repo directory. Run:
    ```
    /fix-issue <issue-number>
    ```
    where `<issue-number>` is the issue created in Check 2 (or any real open issue).

    Verify:
    - Claude reads the issue title and body without you manually copying it
    - Claude applies a fix (or reports it is already fixed)
    - A PR is opened referencing the issue number

    ---

    **Check 4 — CI-04: Fix commit does not re-trigger CI**

    After the PR from Check 3 is created:
    - Navigate to the Actions tab in GitHub
    - Observe that NO new CI run was triggered for the fix branch commit
    - The `[skip ci]` marker in the commit message and GITHUB_TOKEN behavior both prevent this

    ---

    **Check 5 — CI-05: needs-human label after 3 attempts**

    This check can be done with a code inspection rather than 3 actual failed PRs:

    ```bash
    grep -A5 "COUNT\|attempts" .claude/commands/fix-issue.md | head -20
    ```

    Expected: The COUNT >= 3 branch exists, adds `needs-human` label, posts a stop comment, and does not call `gh pr create`.

    For full end-to-end verification (optional): Manually post 3 comments starting with `<!-- fix-attempt -->` on a test issue as any user, then run `/fix-issue <that-issue>` — the command should apply `needs-human` label and stop without opening a PR.

    ---

    Report results for all 5 checks.
  </how-to-verify>
